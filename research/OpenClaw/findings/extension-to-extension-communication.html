What is it?

Extension-to-Extension (E2E) Communication is the mechanism by which multiple OpenClaw extensions can coordinate, share data, and react to shared events through a common EventBus. Since all extensions loaded in the same session share the same EventBus instance, one extension can subscribe to events fired by another extension, enabling indirect communication patterns.

How it works

Core Architecture: Shared EventBus

ExtensionRunner
  ├── createEventBus() → EventBus instance
  ├── extensions: Extension[]
  │     └── each Extension
  │           └── handlers: Map
  └── emit(event) → chains handlers sequentially

All extensions in a session share ONE EventBus instance created in loader.js line 287:

const resolvedEventBus = eventBus ?? createEventBus();

When loadExtensions() is called, the same eventBus is passed to each extension's createExtensionAPI(), where it's stored as pi.events.

Event Emission Pattern

Every emit method in ExtensionRunner follows the same pattern:

async emit(event) {
    const ctx = this.createContext();
    for (const ext of this.extensions) {
        const handlers = ext.handlers.get(event.type);
        if (!handlers || handlers.length === 0) continue;
        for (const handler of handlers) {
            try {
                const handlerResult = await handler(event, ctx);
                if (handlerResult !== undefined) {
                    event.payload = handlerResult; // chain the result
                }
            } catch (err) {
                this.emitError({ extensionPath: ext.path, event: event.type, error: err.message, stack: err.stack });
            }
        }
    }
    return event;
}
Key behavior: If a handler returns a non-undefined value, it becomes the event payload for subsequent handlers (chained transformation). If handler returns undefined, payload passes through unchanged.

Context Object

All handlers receive a shared ctx (ExtensionContext) with:

  • session — current AgentSession
  • model — active model
  • cfg — gateway config (contains secrets!)
  • agentDir — agent directory path
  • runtime — ExtensionRuntime with session-state methods

Uses

  • Inter-extension coordination — Extension A fires a custom event, Extension B reacts
  • Chained transformation — Multiple extensions each transform a payload sequentially
  • Session state sharing — Extensions read/write via shared runtime.appendEntry() to session store
  • Tool result interception — Extension A registers tool, Extension B intercepts result via tool_result
  • Provider request middleware — Multiple extensions each transform API request payload
  • Session lifecycle hooks — Clean up resources when session ends via session_shutdown

Problems

  • No priority ordering — Handler execution order is insertion order (Map iteration), no way to control priority
  • Silent error swallowing — Errors in one handler don't stop others, but also no rollback
  • No dead-letter queue — Failed events are logged via emitError() but no retry mechanism
  • No pub/sub acknowledgment — Publishing extensions have no way to know if any handler processed the event
  • Shared EventBus across all extensions — One misbehaving extension could flood the event bus
  • Session-scoped only — Events don't persist across sessions; new session = fresh EventBus

Solutions

  • Priority via naming convention — Prefix handlers with priority numbers (e.g., 01_myHandler)
  • Error isolation — Each handler wrapped in try/catch individually; failures don't cascade
  • Idempotent handlers — Design handlers to handle duplicate/missing events gracefully
  • Extension ordering at load time — Control which extension loads first to determine handler order
  • State persistence via appendEntry — Use runtime.appendEntry() for cross-handler data sharing

Edge Cases

  • Handler throws synchronous exception — Caught by try/catch, logged via emitError(), event continues to next handler
  • Handler returns undefined — Payload passes through unchanged (not replaced)
  • No handlers for event — emit() returns event unchanged; hasHandlers() returns false
  • Same handler registered twice — Both instances are called (no deduplication)
  • Extension loaded twice — Both instances share same EventBus; both handlers called
  • session_shutdown handler throws — Exception caught, logged, shutdown continues
  • before_provider_request returns wrong type — TypeScript types don't enforce at runtime; downstream provider may receive unexpected payload

Creative Uses

  • Distributed tracing — Each extension appends span info via appendEntry(), collected at session end
  • Event-driven middleware chain — Build a middleware pipeline where each extension inspects/modifies a payload
  • Extension dependency injection — Use resources_discover to announce capabilities other extensions consume
  • Cross-cutting logging — Multiple extensions attach to tool_result/turn_end with different loggers
  • Dynamic feature flags — One extension sets flags via runtime, another reads them in handler
  • Chaos engineering — Test extension that randomly delays or drops events to test error handling
